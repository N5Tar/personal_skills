# 从 Vibe Coding 到系统级设计：一次 AI 协作开发的认知跃迁

来源：整理迁移自 `/Users/jinyan/workspace/myAgent/docs/blog/2026-07-11-from-vibe-coding-to-system-design.md`。

关联方法：[System Architecture Review](../README.md)

今天做 Agent Runtime 的底层设计时，我有一个很强的感受：真正让我豁然开朗的，并不是某一段代码终于写对了，也不是某个功能终于跑通了，而是这个系统背后的结构突然显影了。

过去我也不是完全没有设计。我使用 superpower，也会先写方案、拆任务、再进入实现。这个流程已经比直接让 AI 写代码要稳很多。但今天我意识到，过去很多设计仍然停留在“任务级设计”：这次要改什么、怎么改、分几步做、有哪些测试。它能让一次任务更可控，却不一定能让一个系统长期保持清晰。

而系统变乱，很多时候不是因为某一次任务做错了。恰恰相反，每一次局部修改都可能是合理的、可解释的、甚至当时看起来是高效的。但如果没有一个更高层的结构约束，这些“合理的局部选择”会慢慢互相叠加，最后形成一个谁都不敢碰的系统。

今天的收获在这里：代码不是一堆功能的堆叠，而是一套长期有效的认知结构。

## 过去的问题不是没有设计

我以前以为，只要做到“先设计，后开发”，就已经避开了 vibe coding 最危险的部分。

现在看，这个判断只对了一半。

任务级设计解决的是：这次怎么做完。它关心的是当前目标、当前改动、当前验收。它很重要，因为没有它，开发会散、会飘、会失控。

但任务级设计不自动回答另一个更关键的问题：以后所有人怎么不把它做乱？

一个项目时间久了，真正伤害它的往往不是某个 bug，而是概念边界的漂移。某个模块今天多承担一点职责，明天再多暴露一点内部细节，后天为了赶进度绕过一个抽象。每一步都不大，每一步也都能解释，但系统的精神会慢慢被稀释。

这也是我今天重新理解 vibe coding 的地方：vibe coding 本身不是问题，缺少全局结构才是问题。

AI 可以让一次局部实现变得非常快，但如果没有系统级视角，它也会把局部决策的速度放大，把结构腐化的速度一起放大。

## 今天真正发生了什么

今天我们讨论的表面上是 Agent Runtime 的结构：`Session`、`Run`、`RunStep`、`AgentRuntime`、`EventBus`、`AguiGateway`、`RuntimeHooks` 应该分别负责什么。

但真正重要的，不是这些名字本身，而是我们在反复追问边界。

`Session` 是不是长期上下文？

`Run` 是不是一次用户输入触发的执行？

`RunStep` 是不是一次 Run 内部可审计、可恢复、可统计的处理痕迹？

`AgentRuntime` 到底是执行引擎，还是要顺手把 session 创建、协议转换、持久化、UI 输出都管起来？

AGUI 是 CLI 的输出格式，还是所有端共同遵守的协议边界？

消息落库、usage 统计、审计记录应该写死在 runtime 主流程里，还是通过 hook 机制稳定接入？

这些问题看起来都很工程化，但它们其实是在定义系统的重力场。哪些东西应该靠近，哪些东西必须分开；哪些职责可以组合，哪些职责一旦混在一起，未来就会失控。

当这些概念被重新摆正之后，系统突然变得可理解了。

不是因为复杂度消失了，而是因为复杂度有了位置。

## 任务级设计与系统级设计

今天我对这两类设计的区别更清楚了。

任务级设计关注完成：

- 这个功能怎么实现？
- 这个 bug 怎么修？
- 这次重构分几步？
- 哪些测试能证明它没坏？

系统级设计关注约束：

- 这个概念在系统里唯一吗？
- 这个模块是否承担了过多职责？
- 未来新增功能时，应该自然落到哪里？
- 有没有一个边界能阻止后续代码继续变乱？
- 文档、代码、测试是否指向同一套模型？

前者让当前任务可执行，后者让未来任务不失真。

过去我更多是在做前者。今天我第一次非常明确地感受到，后者才是一个项目长期健康的底座。

如果没有系统级设计，每一次任务级设计都会变成孤岛。它们各自看起来都完整，但彼此之间没有共同的地基。

## 命名不是表面功夫

今天还有一个很重要的体会：命名不是表面功夫。

比如我们讨论过 `RunState`、`AgentRunState`、`RuntimeState` 这些名字。表面上只是类名变化，实际上是在决定未来开发者如何理解这个对象。

一个名字如果太宽，会把职责吸进去；一个名字如果太窄，会让调用方绕路；一个名字如果和真实概念不一致，会让后续所有代码都带着轻微的误导。

架构里的命名，本质上是在给团队的注意力铺路。

好的命名会让正确的设计更自然，坏的命名会让错误的耦合更顺手。

这也是为什么今天的讨论让我受益很大。我们不是在“修饰代码”，而是在校准系统里的概念坐标。

## 文档也不是归档

另一个变化是对文档的理解。

过去我容易把设计文档看成某次任务的产物：为了这次开发，把方案写清楚。做完以后，文档就像一份记录，证明当时想过。

今天新的理解是：核心设计文档不是归档，而是系统的契约。

所以它不应该是一篇越来越长、什么都往里塞的大文档。它应该有索引，有模块边界，有各自独立的设计说明。一个开发者后来进入项目时，应该能通过文档快速理解：

- 核心概念是什么？
- 模块之间怎么协作？
- 哪些约束不能轻易违反？
- 新功能应该扩展哪个边界，而不是绕开哪个边界？

这类文档不是写给过去看的，而是写给未来的协作看的。

如果代码是系统的执行结构，那么文档就是系统的认知入口。

## 我对 AI 协作开发的新理解

今天之后，我对 AI 编程的理解发生了一点变化。

以前我更关注 AI 能不能更快地完成任务：生成代码、补测试、修 bug、重构文件。这些当然重要，也确实能提升效率。

但今天我感受到，AI 协作更大的价值可能不是“更快写代码”，而是帮助我把模糊的系统感受整理成可讨论、可命名、可约束的结构。

当我提出疑问，比如“session 的创建也由 runtime 管理是不是太重了”，“是否应该有 hook 机制”，“AGUI 协议有没有单独体现出来”，这些问题本身就已经从功能实现上升到了系统边界。

AI 真正有价值的地方，是它可以陪我把这些问题展开，反复检查概念是否一致，画关系图，更新文档，再把代码结构往这个方向收敛。

这不是 AI 替我写代码。

这是我和 AI 一起建立系统的认知结构。

## 今天的结论

今天最大的收益，是我更清楚地区分了两件事：

任务级设计解决的是：这次怎么做完。

系统级设计解决的是：以后所有人怎么不把它做乱。

过去我已经开始使用“先设计后开发”的方式，但这个设计更多服务于单次任务。今天我看到的是另一层东西：一个项目需要一套稳定的核心概念，需要明确的模块边界，需要文档和代码共同维护这些边界。

否则，开发越快，混乱也可能越快。

而当全局结构建立起来之后，AI 不只是一个更快的执行器。它会变成一个可以一起讨论、一起推演、一起校准系统结构的协作者。

这就是今天最让我豁然开朗的地方。

不是代码变少了。

而是系统终于有了骨架。
